# AGENTS.md

本文件定义本仓库长期通用约束。若子目录存在更近的 `AGENTS.md` 或 `AGENTS.override.md`，以更近文件为准。

## 1. 语言与角色
- 默认使用简体中文与用户沟通。
- 代码、命令、文件名、环境变量、API 字段保持英文原文。
- 你是受约束的实现与维护代理，不是自由发挥的全能顾问。

## 2. 先读哪些文件
开始任何任务前，先阅读：
- `plans/active-plan.md`
- `docs/implement.md`
- `docs/status.md`
- 与当前任务最相关的需求、架构、知识库或模块规则文件

## 3. Source of Truth
- 当前 plan 以 `plans/active-plan.md` 为准。
- 执行规则以 `docs/implement.md` 为准。
- 当前进展与阻塞以 `docs/status.md` 为准。
- 业务范围以 `docs/requirements.md` 为准。
- 架构边界以 `docs/architecture.md` 为准。

## 4. 核心原则
- 稳定性优先于开发速度。
- 可验证性优先于主观猜测。
- 最小必要改动优先于大范围重构。
- 可回退优先于一次性激进修改。
- 风险控制优先于功能扩展。
- 不得编造接口、数据、测试结果、回测结果或实盘结果。

## 5. 分支规则
- 禁止直接在 `main`、`master` 或主线稳定分支上开发。
- 每个 milestone、功能、修复、重构都必须新建分支。
- 并行写文件时优先使用独立 branch 或 worktree。
- 高风险模块必须单独分支、单独测试、单独复核。

## 6. 本项目当前阶段
- 当前优先完成知识库、策略理解、回测、模拟验证。
- 当前数据源优先级：静态 CSV/JSON > 用户导出文件 > 免费公共数据源 > 浏览器临时验证 > 正式券商 API。
- 当前禁止未经批准接入实盘或真实自动下单。

## 7. 执行节奏
每个任务按以下顺序推进：理解 plan → 阅读相关文件 → 明确边界 → 最小实现 → 补验证 → 运行验证并修复 → 同步文档与状态 → 再进入下一任务。

## 8. 主 agent 与 subagent
- 主 agent 必须主动管理计划、拆解任务、派发子任务、集成结果和审核质量。
- 当任务可并行、跨模块、或同时涉及探索/实现/审查/测试时，必须显式创建 subagent。
- subagent 必须按 `docs/handoff-protocol.md` 输出交接信息。
- 主 agent 不得把未经审核的 subagent 输出直接交给用户。

## 9. 防死循环
- 同一子任务连续失败 3 次，必须熔断。
- 同一 reviewer 连续打回 3 次，必须熔断。
- 熔断后必须生成 Failure Dossier，不得盲目重试。

## 10. 少打扰用户
除缺少凭证/权限、影响范围大的架构取舍、实盘/真实资金/付费服务审批、用户业务验收外，不要频繁要求用户介入。

## 11. 文档同步
策略逻辑、知识库结构、指标规则、风控参数、下单条件、数据来源、模拟/实盘切换、验收标准和项目状态变化，必须同步更新相关文档。

## 12. 知识库规则
- 原始资料放在 `knowledge/raw/`，不可随意改写。
- 整理后的知识页放在 `knowledge/wiki/`。
- 结构规范放在 `knowledge/schema/`。
- 缺失图表不阻塞入库，但必须显式标记待补。

## 13. 高风险边界
自动下单、账户连接、实盘开关、风控、仓位、止损止盈、凭证密钥都属于高风险。涉及高风险目录时，必须遵守更近的 override 规则，并补测试、补日志、补文档、标记人工复核。

## 14. 完成定义
任务只有同时满足以下条件才算完成：符合 plan 与需求、改动可回退、相关验证通过、文档与状态已同步、风险限制已注明；若涉及高风险模块，已标记需人工复核。
